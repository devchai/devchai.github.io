코딩하는 방식이 조금 달라지기 시작했다.

## Codex는 이제 채팅창에 답만 하는 AI가 아니다

OpenAI는 Codex를 여러 작업을 동시에 맡길 수 있는 데스크톱 앱으로 공개했다.

**핵심은 APP이 아니라 병렬 에이전트와 작업 격리(worktree)다.**

---

## 무슨 일이 있었나

OpenAI가 Codex 데스크톱 앱(macOS)을 공개했다.

겉으로 보면 "웹에서 쓰던 Codex를 앱으로 옮긴 것"처럼 보일 수 있다.

**하지만 실제로 바뀐 건 형태가 아니라 작업 방식이다.**

### 기존 Codex 사용 흐름

- 질문한다
- 기다린다
- 답을 받고 복사한다
- 다시 수정 요청한다

항상 하나의 흐름만 다룰 수 있었다.

### Codex 앱은 이 구조를 바꿨다

- 여러 Codex 에이전트를 동시에 실행할 수 있고
- 각 에이전트는 서로 다른 작업을 병렬로 수행한다
- 그리고 이 작업들은 Git worktree로 서로 완전히 분리된다

즉, **한 AI와 대화하는 방식이 아니라 여러 AI에게 일을 나눠서 맡기고 관리하는 구조**에 가깝다.

---

## 왜 이게 중요한가

이번 변화가 흥미로운 이유는 **"Codex가 더 똑똑해졌기 때문"이 아니다.**

**Codex를 사용하는 사람의 역할이 바뀌기 시작했기 때문이다.**

### 기존 AI 코딩 도구

내 옆에서 코드 한 줄씩 도와주는 보조 도구였다.

### Codex 앱의 방향

- 작업을 정의하고
- 여러 에이전트에게 나눠서 맡기고
- 결과를 리뷰하고 합치는 쪽에 더 가깝다

이 구조에서는 **개발자가 키보드를 치는 시간보다 작업을 나누고 판단하는 시간이 늘어난다.**

---

## 병렬 에이전트 + worktree, 이게 핵심이다

Codex 앱의 핵심은 두 가지다.

### 1. 병렬 에이전트 (Parallel Agents)

하나의 요청을 기다릴 필요가 없다.

예를 들면 이런 식이다:

- **에이전트 A**: 로그인 UI 컴포넌트 작성
- **에이전트 B**: 인증 API 연동
- **에이전트 C**: 테스트 코드 작성

이 작업들이 동시에 진행된다.

개발자는 각 작업의 진행 상황과 결과만 확인하면 된다.

### 2. 작업 격리 (Git worktree)

여기서 중요한 게 **worktree**다.

각 에이전트는 같은 저장소를 쓰지만 **서로 다른 작업 공간**에서 작업한다.

그래서:
- 코드가 꼬이지 않고
- 브랜치 충돌도 줄고
- 결과를 비교하고 합치기도 수월하다

단순히 "여러 창을 띄운 것"과는 다르다.

---

## 누가 특히 체감할까

### 시니어 개발자 / 테크 리드

- 구현 자체보다 구조와 리뷰에 집중하고 싶은 사람
- 반복 작업을 덜고 싶은 사람

### 1인 개발자 / 소규모 팀

- 프론트, 백엔드, 테스트를 동시에 굴려야 하는 상황
- 병렬 작업의 체감이 크다

### PM / 기획자

- "이거 가능해요?"를 묻기 전에
- 간단한 프로토타입을 바로 확인해보고 싶은 경우

---

## 써본다면 이렇게 접근하는 게 좋다

Codex 앱은 **"한 번에 큰 일을 시키는 도구"라기보다는 일을 잘게 나눌수록 효과가 커지는 도구**다.

- 큰 기능 하나 ❌
- 작은 작업 여러 개 ⭕

그리고 결과는 그대로 쓰기보다는 **리뷰하고 수정하는 전제**로 보는 게 맞다.

결국 중요한 건 **AI보다 사람 쪽의 작업 설계 능력**이다.

---

## XSignal의 시그널

이번 Codex 앱은 **"AI가 코딩을 얼마나 잘하느냐" 경쟁의 연장이 아니다.**

그보다 더 중요한 신호는 이거다:

> **AI 코딩 도구의 경쟁 무대가**  
> **생성 능력 → 협업 구조와 워크플로우로 옮겨가고 있다.**

앞으로는 프롬프트를 잘 쓰는 사람보다 **AI에게 일을 잘 나누는 사람**이 더 편하게 개발할 가능성이 크다.

Codex 앱은 그 방향을 꽤 명확하게 보여준 사례다.

이 변화는 "AI가 코드를 잘 쓰는가"를 넘어 **"AI+개발팀의 협업 구조"에 초점**을 맞추게 한다.
